# MemoryMesh Agent Template

`MemoryMesh`는 Codex, Claude Code, Antigravity CLI, OpenCode 같은 여러 코딩 에이전트가 같은 프로젝트 폴더에서 작업 맥락을 공유하고 작업을 이어받게 돕는 파일 기반 협업 메모리입니다. 한 에이전트가 토큰 한도에 다다랐을 때 다른 에이전트가 맥락을 그대로 인계받아 계속하는 것이 주목적입니다.

별도의 서버, 데이터베이스, 플러그인 없이 프로젝트 폴더 안의 Markdown 파일만으로 작동합니다.

## 핵심 개념

여러 에이전트를 번갈아 쓰다 보면 다음 문제가 자주 생깁니다.

- 이전 에이전트가 무엇을 했는지 새 에이전트가 모릅니다.
- 같은 프로젝트를 다시 분석하느라 컨텍스트를 많이 씁니다.
- 토큰 한도에 도달하면 작업 상태가 끊깁니다.
- Codex, Claude, Antigravity(Gemini), OpenCode 사이에서 설명을 직접 복사해 붙여야 합니다.
- 작업 중간 결정, 변경 파일, 남은 작업이 흩어집니다.

`.mesh`는 이 문제를 줄이기 위해 프로젝트 안에 공유 작업 기억을 둡니다.

```text
.mesh/
├─ PROTOCOL.md      모든 에이전트가 따르는 공유 규칙
├─ OVERVIEW.md      이어받는 에이전트용 현황 브리프
├─ work.log         append-only 작업 이벤트 로그
├─ INDEX.md         오래된 기록을 찾는 색인
├─ handoffs/        에이전트 간 인계 파일
├─ archive/         로테이션된 오래된 로그
└─ cold/            장기 요약
```

에이전트는 작업 전에 `.mesh`를 읽고, 작업 후 `work.log`에 기록하며, 다른 에이전트가 이어받아야 할 때 `handoffs/`에 인계 파일을 남깁니다.

## 가볍게 시작하고 필요할 때 확장

MemoryMesh는 처음에 인계 연속성에 필요한 구조만 만듭니다. 개념 지식을 모으는 영역은 처음부터 만들지 않습니다. 같은 개념을 여러 세션에 걸쳐 반복 설명하거나, 결정 기록이 쌓여 한눈에 보기 어려워지는 신호를 에이전트가 감지하면 그때 사용자에게 추가를 제안합니다. 사용자는 확장 시점을 판단할 필요 없이 승인 여부만 답하면 되고, 승인하면 기존 구조를 건드리지 않고 해당 폴더가 더해집니다.

확장으로 더해지는 영역은 다음과 같습니다.

- `context` 프로젝트 전체에 적용되는 고정 전제와 공통 규칙. 자주 바뀌지 않는 배경 지식.
- `wiki` 안정적으로 재사용하는 개념 지식. 파일 사이를 `[[파일명]]` 백링크로 연결합니다.
- `graph` 항목 사이의 관계(영향, 참조, 모순 등)를 기록하는 데이터.
- `decisions` 주요 의사 결정의 배경과 이유를 모은 로그.

확장한 뒤의 구조는 다음과 같습니다.

```text
.mesh/
├─ PROTOCOL.md      공유 규칙
├─ OVERVIEW.md      현황 브리프
├─ work.log         작업 이벤트 로그
├─ INDEX.md         색인
├─ handoffs/        인계 파일
├─ archive/         오래된 로그
├─ cold/            장기 요약
├─ context/         고정 전제와 공통 규칙   (확장 시 추가)
├─ wiki/            개념 지식과 백링크       (확장 시 추가)
├─ graph/           항목 간 관계             (확장 시 추가)
└─ decisions/       의사 결정 로그           (확장 시 추가)
```

확장하면 메모리가 두 층으로 작동합니다. 하나는 무엇을 했고 누가 이어받는지를 다루는 작업 이력층(work.log, handoffs)이고, 다른 하나는 무엇을 아는지를 다루는 지식 그물망층(wiki, graph)입니다. wiki의 백링크와 graph의 관계가 서로 얽혀 그물망을 이루는데, MemoryMesh라는 이름이 바로 이 구조에서 나왔습니다. 인계가 잦은 프로젝트에서 같은 배경 지식을 매번 다시 설명하지 않고 이 그물망을 참조하게 하는 것이 확장의 목적입니다.

## 각 파일의 역할

| 파일 또는 폴더 | 역할 |
|---|---|
| `memorymesh-setup.md` | 작업 폴더에 두고 에이전트에게 적용을 지시하면, 신규 폴더에는 `.mesh` 구조를 새로 만들고 기존 폴더에는 기존 지침 파일을 보존하며 병합하는 설치 프롬프트 |
| `AGENTS.md` | 지침 본문(관리 블록)을 담는 진입점. Codex와 OpenCode가 읽습니다. |
| `CLAUDE.md` | `AGENTS.md`를 가리키는 포인터. Claude Code가 읽습니다. |
| `GEMINI.md` | `AGENTS.md`를 가리키는 포인터. Antigravity CLI가 읽습니다. |
| `.mesh/PROTOCOL.md` | 모든 에이전트가 따를 공유 작업 규칙 |
| `.mesh/OVERVIEW.md` | 새 에이전트가 프로젝트를 빠르게 이해하기 위한 요약 |
| `.mesh/work.log` | 최근 작업을 남기는 append-only 로그 |
| `.mesh/INDEX.md` | 과거 로그, 주제, 핸드오프 파일을 찾는 색인 |
| `.mesh/handoffs/` | 다른 에이전트에게 넘길 인계 파일 보관 |
| `.mesh/archive/` | 오래된 상세 로그 보관 |
| `.mesh/cold/` | 장기 요약, 월별 요약본 보관 |

지침 본문은 `AGENTS.md` 한 곳에만 두고 `CLAUDE.md`와 `GEMINI.md`는 이를 링크해 활용합니다. 규칙을 한 군데서 관리하기 위한 것입니다. OpenCode는 별도의 `opencode.md`를 표준 지침으로 읽지 않고 `AGENTS.md`를 읽으므로, `opencode.md`는 없습니다.

루트 지침 파일(`AGENTS.md`, `CLAUDE.md`, `GEMINI.md`)과 `.mesh` 구조는 설치할 때 `memorymesh-setup.md`가 만들어 줍니다. 따라서 배포에 꼭 필요한 파일은 이 README와 `memorymesh-setup.md` 두 가지입니다.

## 요청 처리 규칙

에이전트는 요청을 받으면 그 성격을 먼저 나눈 다음 행동합니다.

### 쿼리 모드

정보만 요구하는 경우입니다. 예를 들면 이렇습니다.

- "핸드오프 파일이 몇 개야?"
- "OVERVIEW.md에 스택이 뭐라고 쓰여 있어?"
- "현재 상태가 어때?"
- "이 접근이 괜찮아 보여?"

이때 에이전트는 답에 직접 필요한 파일만 읽고, 세션 시작 루틴을 건너뛰며, 답을 보고하고 멈춥니다. 다음 행동을 예단하지 않고 사용자가 정할 때까지 기다립니다.

### 상태와 의견 규칙

사용자가 현황, 진행, 의견, 평가를 물으면 관찰한 사실이나 평가만 답하고 멈춥니다. 다음 행동으로 자동으로 넘어가지 않고, 사용자가 계속 작업하기를 원한다고 가정하지 않습니다. 이 규칙은 에이전트가 상태 질문에 답한 뒤 승인받지 않은 변경이나 작업을 멋대로 시작하는 패턴을 막습니다.

### 질문과 명령을 가른다

의문문은 질문이고, 명령문이라야 실행입니다. 질문에 든 동사(지우다, 바꾸다, 옮기다, 실행하다 등)는 확인 대상이지 실행 명령이 아닙니다. 예를 들어 "A 파일이 지워져 있나?"는 존재 여부를 확인해 그렇다 또는 아니다로 답할 질문입니다. 이때 에이전트는 A를 지우지 않고, 지울지 되묻지도 않습니다. 되묻는 것도 질문을 행동의 출발점으로 잘못 분류한 신호이기 때문입니다. 에이전트의 특성상 이런 워크플로가 항상 보장되는 것은 아닙니다. 수시로 이 지침을 어길 수도 있다는 점을 고려할 필요는 있습니다.

### 읽기와 변경의 경계

답하거나 맥락을 파악하기 위한 읽기(파일 존재 확인, 내용 보기, 목록 조회)는 권한을 묻지 않고 바로 수행합니다. 즉, 읽기를 하겠다고 권한을 요청하지 않습니다. 요청이 삭제 같은 파괴적 작업의 권한 요청으로 오해될 수 있기 때문입니다. 권한 확인과 명시적 명령은 변경 작업(쓰기, 삭제, 이동, 실행)에만 적용하고, 변경은 명시적 명령이 있을 때만 수행합니다.

### 태스크 모드

사용자가 생성, 수정, 삭제, 이동, 실행을 명령문으로 요구하는 경우입니다. 이때만 세션 시작 루틴(`PROTOCOL.md`, `OVERVIEW.md`, `work.log` 최근 꼬리 읽기)을 수행한 뒤 작업합니다.

## 설치 방법

MemoryMesh는 압축 파일을 풀어 넣는 방식이 아니라, 설치 프롬프트 파일(`memorymesh-setup.md`)을 작업 폴더에 두고 에이전트에게 적용을 지시하는 방식으로 설치합니다. 절차는 신규 폴더든 기존 폴더든 동일합니다.

1. 사용하려는 에이전트(Codex, Claude Code, Antigravity CLI, OpenCode 중 하나)를 원하는 작업 폴더에서 엽니다.
2. 그 폴더에 `memorymesh-setup.md`를 둡니다.
3. 에이전트에게 이렇게 요청합니다. 정확하게 같은 표현이 아니라도 이와 같은 취지로 명령을 작성하면 됩니다.

```text
memorymesh-setup.md 파일의 설정 내용을 이 폴더에 적용하라.
```

`memorymesh-setup.md`는 먼저 폴더의 현재 상태를 점검한 뒤 상황에 맞게 동작합니다.

- 신규(빈) 폴더라면 `.mesh` 구조와 루트 지침 파일(`AGENTS.md`, `CLAUDE.md`와 `GEMINI.md`)을 알아서 새로 만듭니다.
- 기존 폴더라면 이미 있는 `AGENTS.md`, `CLAUDE.md`, `GEMINI.md`나 다른 설정 파일을 덮어쓰지 않고 보존하면서, 빠진 `.mesh` 구조와 관리 블록만 병합합니다.

따라서 사용자가 신규인지 기존인지 구분힐 필요가 없습니다. 같은 프롬프트가 두 경우를 모두 처리하며, 같은 폴더에 다시 적용해도 안전합니다. 불안하다면 직접 확인해 처리하는 것이 좋습니다.

설치한 뒤에는 현황을 초기화합니다.

```text
이 프로젝트의 .mesh 구조를 확인하고 OVERVIEW.md를 현재 프로젝트 기준으로 초기화하라.
```

이제 일반 작업을 요청하면 됩니다.

```text
간단한 주사위 게임을 만들어줘. 끝나면 Claude에게 넘겨라.
```

신규 폴더에 설치한 직후의 구조는 다음과 같습니다.

```text
dice-game/
├─ AGENTS.md
├─ CLAUDE.md
├─ GEMINI.md
└─ .mesh/
   ├─ PROTOCOL.md
   ├─ OVERVIEW.md
   ├─ work.log
   ├─ INDEX.md
   ├─ handoffs/
   ├─ archive/
   └─ cold/
```

루트의 `AGENTS.md`와 포인터 파일들이 `.mesh/PROTOCOL.md`를 참조하므로, 에이전트는 작업 전후로 `.mesh`를 활용하도록 안내됩니다.

## 기본 작업 흐름

```text
1. Codex가 기능 구현
2. Codex가 .mesh/work.log에 작업 기록
3. Codex가 Claude용 handoff 파일 생성
4. Claude가 .mesh와 handoff 파일을 읽고 보완
5. Claude가 .mesh/work.log에 결과 기록
6. Gemini가 UX 문구나 설명을 검토
7. Gemini가 .mesh/work.log와 필요한 handoff를 기록
```

사용자는 매번 이전 작업을 길게 설명할 필요가 없습니다. 예를 들어 Codex에서 먼저 요청합니다.

```text
간단한 주사위 게임을 만들어줘. 끝나면 Claude에게 넘겨라.
```

Codex는 작업 후 다음을 남깁니다.

```text
.mesh/work.log
.mesh/handoffs/handoff_dice-game.<model-id>.md
```

이후 Claude Code에서 이어받습니다.

```text
.mesh를 확인하고 Codex가 남긴 주사위 게임 작업을 이어서 보완하라.
```

## 체크포인트와 컨텍스트 압력 관리

긴 작업 도중 상태를 저장하도록 설계되어 있습니다. 에이전트는 다음 상황에서 `CHECKPOINT` 이벤트를 `.mesh/work.log`에 남깁니다. 명백히 유용한 체크포인트는 묻지 않고 바로 기록합니다.

- 파일 3개 이상을 수정했거나 파일을 생성, 이동, 삭제한 경우
- 구현, 테스트, 리팩터링, 문서화로 작업 단계가 길어진 경우
- 많은 파일을 살펴 작업 맥락이 커진 경우
- 사용자가 작업 도중 방향을 바꾼 경우
- 다른 에이전트가 이어받을 가능성이 있는 경우

예시 로그는 다음과 같습니다.

```text
### 2026-06-28 16:20 | claude-opus-4-8 | CHECKPOINT
Summary: 주사위 게임의 기본 구조, 점수 시스템, 주사위 굴리기 로직을 구현함.
Changed files: index.html, style.css, game.js
Current focus: 승패 판정과 재시작 흐름 보완 중.
Remaining: 게임 설명 문구, 애니메이션 효과, 코드 정리.
Risks: 작업 맥락이 길어지고 있어 다음 작업 전 handoff 작성 권장.
```

컨텍스트가 무거워졌다고 판단하면 에이전트는 `CONTEXT_PRESSURE` 이벤트를 남기고, 다른 에이전트나 새 세션으로 넘기도록 handoff 파일 생성을 권고할 수 있습니다.

## 종료 전 상태 저장

작업을 끝내기 전에 현재 상태를 저장해 두면 다른 에이전트나 새 세션에서 이어받기 쉽습니다. 종료 전에 이렇게 요청합니다.

```text
종료 전 .mesh에 현재 상태를 저장하고, 다른 에이전트가 이어받을 수 있도록 정리하라.
```

에이전트는 현재 상태 요약, 완료 작업, 변경 파일, 남은 작업, 위험 요소를 정리하고 `.mesh/work.log`에 `CHECKPOINT`, `WORK_END` 또는 `HANDOFF`를 기록한 뒤 필요하면 handoff 파일을 만듭니다. 정리가 끝나면 실제 종료 명령을 실행하면 됩니다.

## 메모리 계층 구조

MemoryMesh는 세 단계 메모리를 씁니다.

| 계층 | 위치 | 용도 | 읽는 빈도 |
|---|---|---|---|
| Hot | `.mesh/work.log` | 최근 작업 기록 | 매 작업 |
| Warm | `.mesh/archive/` | 오래된 상세 로그 | 필요할 때 |
| Cold | `.mesh/cold/` | 장기 요약 | 명시적으로 필요할 때 |

목표는 에이전트가 매번 모든 과거 기록을 읽지 않게 하는 것입니다. 최근 기록은 `work.log`에서 확인하고, 오래된 내용은 `INDEX.md`로 필요한 파일만 찾아 읽습니다.

## 주의 사항

이 템플릿은 강제 실행 프로그램이 아니라 파일 기반 협업 규칙입니다. 에이전트가 규칙을 지키지 않으면 사용자가 직접 확인해야 할 수 있습니다.

```text
이번 작업을 MemoryMesh 프로토콜에 따라 .mesh/work.log에 기록하고 필요하면 handoff 파일도 만들어라.
```

`.mesh` 폴더 이름 앞에 마침표(`.`)를 둔 이유는 Windows 탐색기, VS Code 탐색기, macOS Finder에서 기본적으로 숨겨지거나 아래로 밀려 프로젝트 파일과 시각적으로 분리되게 하기 위한 것입니다. `.git`, `.vscode` 같은 도구 관례와도 맞습니다. 다만 숨김 폴더라 일부 에이전트의 검색 도구가 건너뛸 수 있어, 프로토콜은 그럴 때 경로를 직접 지정해 읽도록 안내합니다. Windows에서는 탐색기로 직접 만들 수 없으므로 PowerShell이나 터미널에서 생성합니다.

`.mesh`에는 프로젝트 맥락, 변경 내용, 의사 결정, 작업 이력이 들어갑니다. 공개 저장소에서 민감한 프로젝트를 다룬다면 `.gitignore`에 추가하는 것을 권장합니다.

```gitignore
.mesh/
```

단, 템플릿 자체를 공개하는 저장소라면 빈 초기 상태의 `.mesh`는 포함해도 됩니다. 실제 프로젝트 기록이 담긴 `.mesh`를 공개 저장소에 올리지 않도록 주의합니다.

## 이 템플릿이 맞는 경우

- Codex, Claude Code, Antigravity CLI, OpenCode를 번갈아 쓰는 경우
- 한 프로젝트를 여러 세션에 걸쳐 작업하는 경우
- 토큰 한도 때문에 에이전트를 바꿔야 하는 경우
- 작업 기록, 결정, 핸드오프를 파일로 남기고 싶은 경우
- 새 에이전트가 전체 프로젝트를 처음부터 다시 분석하는 일을 줄이고 싶은 경우
- 구현, 리뷰, 테스트, 문서화를 서로 다른 에이전트에 나누어 맡기고 싶은 경우

## 이 템플릿이 맞지 않는 경우

- 한 번 쓰고 버릴 임시 작업
- 파일 시스템 접근이 없는 채팅형 LLM만 쓰는 경우
- 한 에이전트가 짧은 작업을 단독으로 끝내는 경우
- 프로젝트 기록을 파일로 남기면 안 되는 보안 환경

## 빠른 요약

작업 폴더에서 에이전트를 열고 `memorymesh-setup.md`를 그 폴더에 두면, 신규든 기존이든 한 번의 요청으로 설치됩니다.

```text
memorymesh-setup.md 파일의 설정 내용을 폴더에 적용하라.
```
적용 완료 후에는 이 설정 파일은 삭제해도 됩니다.

작업 도중에는 가끔 진행 상황을 정리합니다.

```text
현재까지 진행 상황을 .mesh에 정리하라.
```

종료 전에는 상태 저장을 요청합니다.

```text
종료 전 .mesh에 현재 상태를 저장하고, 다른 에이전트가 이어받을 수 있도록 정리하라.
```

다른 에이전트는 실행 즉시 알아서 같은 폴더에서 `.mesh`를 읽고 작업 이력을 이어받습니다.

```text
.mesh를 확인하고 이전 에이전트가 남긴 작업을 이어서 진행하라.
```
굳이 이어서 하라는 요구를 하지 않고 바로 추가 작업을 진행하면 됩니다.
